Setting the Stage

Title: The Road Ahead for AI in Fabric

Welcome everyone. Today we are going to talk about the road ahead for AI in Microsoft Fabric, specifically data agents and how they are creating intelligent solutions on top of your enterprise data. This is a story about making AI work with the data you already have, not in some disconnected silo, but right where your business runs.

Microsoft Fabric Platform Overview

2️⃣ Act 2: Data Agents Deep Dive

Data Agents Deep Dive

🎞️ Slides 5–21 ⏱️ ~18 min The core of the talk

Start with the "why" (business value), then show the ecosystem (integrations), demonstrate it live (demo), explain what you need to build one, walk through the creation flow, then hit the GA announcement hard with all the new features.

Slide 5

Fabric Data Agents: Virtual Analyst

A Fabric Data Agent is a virtual analyst purpose-built for your data domain. Not a generic chatbot. It understands your tables, business logic, terminology. Point it at your data, give it instructions, and it becomes an expert. It answers natural language questions by generating SQL, reasoning over relationships, and synthesizing insights.

Slide 6

Why Build Data Agents?

Because you need to ground your AI solutions in enterprise knowledge. The hard part is connecting AI to real, governed, trusted data. Fabric Data Agents act as the bridge. Developers, end-users, and other agents can all consume them through Fabric, Copilot Studio, AI Foundry, or a simple API endpoint. Build once, consume everywhere.

Slide 7

Why Data Agents? (Expanded)

Grounding data insights on enterprise knowledge. Whether consumers are end-users asking questions in Teams, developers building custom apps, or other AI agents in a multi-agent architecture — Data Agents provide a consistent, secure, governed way to access enterprise data. One agent, many surfaces.

Slide 8

Architecture: Grounding AI Agents on Fabric IQ

At the center: Fabric IQ (intelligence layer) and your Data Agent with default tools and skills for data retrieval. Left: enterprise data in OneLake. Right: all consumption surfaces. Key concept: context engineering — creating shared curated views through custom skills and tools.

3️⃣ Act 3: Under the Hood

Under the Hood

🎞️ Slides 22–26 ⏱️ ~7 min For the builders

Shift gear to technical. Show the architecture, walk through an actual query flow, explain components, then go deep on MCP. This is where the data engineers lean in.

Slide 22

"How Does It Work?" — Section Divider

Now let us go under the hood and look at how Data Agents actually work.

Slide 23

Technical Design

The Data Agent sits in the middle with tools, context, and configuration. Connects to all data in OneLake on one side, published through REST/MCP on the other. Design principles: open and compatible, support any data in Fabric, built on Azure AI Agents and OpenAI.

💡 The "don't be weird" line always gets a laugh. Lean into it. Philosophy: follow industry standards and shape them where needed.
Slide 24

Data Agent Flow

User asks "how many customers bought X in the last month?" → Agent assesses context and tools → invokes NL2SQL → generates and runs SQL query → returns: 2.3 million customers. Every step is traceable and auditable.

Slide 25

Components

Three categories: Prompts — secret sauce and AI instructions. Tools — data sources and custom tools via MCP. Context — chat history, database logs, and other signals. Accessible via Chat API and MCP Server.

Slide 26

MCP in Data Agents

Every Data Agent is both an MCP Server (invoked as a tool by other agents) and an MCP Client (every data source is a tool). Future: bring your own MCP Servers. This dual role makes Data Agents incredibly composable in multi-agent architectures.

🎯 Data Agents are MCP-native. They are both producers and consumers of tools.

❓ Anticipated Q&A

"Is RLS/CLS enforced when calling via MCP or API?"
Yes. Security permissions, including row-level and column-level security, are always enforced regardless of the consumption surface — whether it is Teams, Copilot Studio, AI Foundry, MCP, or the public endpoint.
"Can I use Data Agents with data outside Fabric?"
Yes, through Mirrored Databases. You can mirror data from Oracle, Snowflake, SQL Server, Databricks, Azure SQL, and more into OneLake and build Data Agents on top. The agent treats mirrored data just like any native Fabric data.
"What LLM powers the agent?"
Data Agents are built on Azure AI Agents and OpenAI. The specific model is managed by the platform. The focus is on the enterprise-grade tooling, governance, and integration rather than letting users swap models.
"When will Code Interpreter be available?"
It is on the roadmap for May 2026 onwards. Sign up for the private preview at aka.ms/DA-private-preview to get early access.
"Can Data Agents write back to the database?"
No. By design, the SQL tool is read-only. This is intentional for security and auditability — the agent can query and reason over data, but cannot modify it.
"How is this different from Copilot in Power BI?"
Copilot in Power BI is optimized for the BI consumption experience — asking questions about reports and visuals. Data Agents are broader — they work across any data in OneLake, generate SQL, support multi-source reasoning, and can be consumed programmatically via API/MCP. Think of Data Agents as the engine that can power Copilot in Power BI and much more.
"What about cost / licensing?"
Data Agents consume Fabric capacity units (CUs). As a GA feature, it is included in your Fabric capacity. Specific CU consumption depends on query complexity and data volume.
"Can I bring my own MCP tools?"
Not yet, but it is on the roadmap. Today, Data Agents use built-in tools (NL2SQL, etc.). BYO MCP Server with custom tools is listed as a future capability.
